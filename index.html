Technical landscape

Scales with your data. Bends with your choices. Belongs entirely to you.

For the technically-minded in the room: here's what's under the hood, and why this architecture stays yours - to grow, to change, to own. Hover a layer to inspect its role and the tools it runs on.

Eddy · the master data module in the diagram above

Eddy - the Eddytor platform in our delivery stack - is a specialized external governance UI that connects to your own cloud storage (ADLS Gen2, Azure Blob, Microsoft Fabric, S3) via encrypted credentials. Your data never leaves your tenant: Eddytor reads and writes in place, acting as a control layer on top of storage you already own rather than a separate data store. It auto-discovers existing tables, enforces schema on every write, supports zero-downtime column additions (backward-compatible by design), and applies data-quality constraints at three levels - fixed domains, dependent fields, and cross-table lookups. Every change is captured in a full transaction history with time-travel and one-click undo. An AI-assisted review layer flags anomalies, duplicates, and missing values for stewards, with pluggable providers (Anthropic, OpenAI, Google, Mistral, or self-hosted Ollama when inference must stay fully local). Access is four-level RBAC - Admin, Builder, Editor, Viewer. And because the contract between Eddy and the Gold layer is just a Delta table, swapping Eddytor for your own MDM tool never breaks the rest of the stack.

Layer contracts

What each boundary guarantees to the next

The contracts that make the "swap any tool" promise real. Replace Auto Loader with Fivetran and only the first boundary changes - everything downstream stays intact because the envelope stays the same.

Boundary
What it guarantees
Enforced by
Source → Bronze
Raw files land with a standard envelope: source, file path, ingest timestamp, checksum. No schema enforcement - the contract is "faithful copy with full traceability."
Auto Loader · Unity Catalog volumes
Bronze → Silver
Schema enforced - column names, types, nullability. Records that fail expectations are quarantined with a rejection reason. Valid records carry forward all Bronze envelope fields.
Spark Declarative Pipelines expectations · schema enforcement
Silver → Gold
Every fact row resolves to a canonical product and customer from master data. Business logic applied, KPIs computed. Unresolvable records routed to Eddy for steward review - never silently dropped.
SQL models · Unity Catalog lineage · Eddy queue
Gold → Consumption
Governed views with row-level and column-level security. Consumers connect via SQL, REST, or DirectQuery. Definitions are frozen at this boundary - no ad-hoc joins back to raw tables.
Unity Catalog grants · Databricks SQL warehouse ACLs
Why this matters. Contracts are what keep components swappable. Replace one tool at one boundary - everything upstream and downstream keeps working because the shape of what flows across the boundary hasn't changed.
Example trace gold.kpi_promo_share ← gold.fact_sales_out ← silver.nielsen_validated ← bronze/nielsen/w15.csv
Every Gold KPI traces back through Unity Catalog lineage to the exact Bronze file, load timestamp, and checksum - queryable via the Unity Catalog API. An auditor can walk from any report number to its source delivery in under a minute.

Shared Unity metastore · Azure Entra ID · shared product hierarchy
Cross-BU aggregation reads from governed views across catalogs. BU-level data is never copied - the PortCo layer queries in place.

Compute isolation. Each BU catalog binds to its own serverless SQL warehouse or job cluster. A runaway query in one BU cannot consume another BU's compute budget.

Data isolation. Unity Catalog enforces catalog-level access grants via Entra ID groups. A DK analyst cannot query SE tables unless explicitly granted cross-catalog access by the PortCo admin.

Optional network isolation. For BUs with stricter requirements, workspaces can be deployed into separate VNets with Private Link - traffic never touches the public internet.

Agentic monitoring · add-on deep dive

How the self-healing loop actually works

Not magic. The agent detects, diagnoses, and proposes - humans approve before anything touches your pipelines. Every action is logged, every fix reviewed.

STEP 01
Detect
SDP event log fires a failure or quality breach
→
STEP 02
Diagnose
Agent inspects logs, schema diffs, source changes, historical patterns
→
STEP 03
Propose
Drafts a fix and a plain-language explanation
→
STEP 04
Approve
Steward reviews in Teams or Slack · one-click approve or override
→
STEP 05
Heal
Approved fix applied, pipeline re-runs, downstream unblocked

Read-only by default. The agent has read-only access to pipeline code, schemas, and logs. It cannot execute changes directly - proposed fixes are always written to a review queue for human approval.

What it can fix autonomously (after approval): schema evolution (promote rescued columns), re-run failed jobs with adjusted parameters, update quarantine rules for known bad-data patterns, restart stalled streaming queries.

What it always escalates: data loss, security-relevant changes, business logic updates, and anything it cannot diagnose with high confidence.

Can we plug in our existing tools?
Yes. Unity Catalog exposes ODBC, JDBC, and REST. Any tool that speaks SQL or REST can read the data - existing notebooks, BI tools, Python scripts, custom apps - without modification.

How does this evolve without breaking my stack?
The Accelerator is versioned. Upgrade engagements pull new capabilities into your workspace deliberately - not automatically. Layer contracts stay stable, so upgrades add features rather than force migrations.
Versioned · Opt-in upgrades
⟳
What happens when a source changes format?
Auto Loader runs in schema evolution mode. New columns are captured in a rescued-data column, the pipeline keeps running, and an alert routes to the steward (or Agentic Monitoring) for promotion review. No 3 AM failure.
Auto Loader · Zero-downtime
✓
What's the testing strategy?
Three layers before anything reaches production. Unit tests on SQL model logic, integration tests on a staging catalog with synthetic data, and data quality expectations on every production refresh. Breaches trigger alerts immediately.
Unit · Integration · DQ
♺
What's the recovery story?
Every layer is idempotent and replayable. Bronze is append-only - a full re-ingest rebuilds everything downstream. Delta time-travel gives point-in-time recovery up to retention. For cross-region DR, geo-redundant storage and a standby workspace.
Idempotent · Time-travel
